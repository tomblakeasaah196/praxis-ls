1.  Executive summary
SmartLS is a genuinely capable first-generation system. It already digitises the core of a freight and logistics business — turning a quote into a costing, a costing into a proforma, an operation into milestones, and a delivery into an invoice and a receivable — and it layers on modern touches that most local competitors do not have: optical character recognition (OCR) for document intake, an AI assist desk, a margin/pricing engine, and a document vault.
It is also, in its current form, fragile and non-compliant. The review found database credentials written in plain text inside the source code, a hidden “God Mode” screen built for hard deletions, server error logs sitting inside web-reachable folders, the same screens copy-pasted five times (once per department), and several duplicated database tables that quietly split the same data into two places.
Two gaps matter most. First, there is no accounting layer at all — no chart of accounts, no general ledger, no journals, no tax engine. The system records invoices and payroll, but it cannot produce the OHADA / SYSCOHADA financial statements that Cameroon law requires, which means the company cannot properly file its taxes (DSF, TVA, CNPS) from its own system. Second, the back-office ERP is English only, while Cameroon is officially bilingual and statutory accounting output is in French.

6.  The headline gap — OHADA accounting & tax
This deserves its own section because it is the single biggest risk and the main reason for the revamp. The current system has financial records but no accounting. It stores invoices, receipts, purchase orders and payroll, but there is no chart of accounts, no double-entry general ledger, no journals, no fiscal-period close and no tax engine. Money movements are tracked as operational documents, never posted to accounts.
In Cameroon that is not a nice-to-have. Companies keep their books under the revised SYSCOHADA framework (AUDCIF) and must produce, file and defend statutory financial statements and tax declarations:
Figures (CIT 33% / 27.5%, minimum tax 2.2% / 5.5%, TVA 19.25%, CNPS rates, DSF deadlines) are current as of June 2026 and should be confirmed against the DGI Code Général des Impôts at build time, since Finance Acts adjust them yearly.

8.  Operating context & constraints
Jurisdiction: Cameroon — bilingual (EN/FR), OHADA/SYSCOHADA accounting, DGI tax administration (DSF, TVA, CIT, CNPS).
Target platform: Node.js back end, React + Vite front end, delivered as an installable PWA, fully bilingual.
Hardware budget: A single server with 12 GB RAM, 6 CPU cores and 75 GB SSD. Generous for a well-built Node + MySQL app at SME scale, but it forces discipline: lean services, off-box file storage and backups, and a clear scaling path.
Data: An existing 84-table database with real operational history that must be migrated, not discarded.

Why this is existential
You cannot file taxes you cannot account for. Without a SYSCOHADA-compliant ledger feeding the DSF, TVA and CNPS declarations from one source of truth, filings are manual, late-prone and hard to defend in an audit. Re-introducing OHADA accounting and reporting is the central requirement of the revamp, not an add-on.
Today's flaw | What the revamp introduces
No accounting / cannot file taxes | A native double-entry OHADA/SYSCOHADA core: chart of accounts, general ledger, journals, period close, and a tax engine that produces TVA, CIT/minimum-tax and the DSF pack — plus CNPS — from one ledger.
ERP English-only | A single internationalisation layer so every screen, document and notification renders in English or French, per user and per document, reusing the website's existing translations.
Hard-coded secrets | Configuration and secrets moved to environment variables / a secrets store, rotatable without code changes.
God Mode hard deletes | Replaced by maker-checker approvals, soft-delete with retention, and a tamper-evident audit trail.
Screens copied per role | One screen, one code path, with a central permission policy deciding visibility and actions — built once, governed centrally.
Permissions in three places | A single RBAC policy engine (roles + capabilities + field-level rules) as the one source of truth for who does what and who sees what.
Duplicate / twin tables | A consolidated, normalised schema migrated cleanly from the current 84 tables, with duplicates merged and de-duplicated.
Server-rendered monolith | A clean API/front-end split: the React PWA talks to a documented Node API, so UI and logic evolve independently.
No offline use | PWA caching and a sync model so key field tasks keep working through patchy connectivity.
Single server, no DR | An explicit hosting, backup and disaster-recovery design sized to the 12 GB / 6-core / 75 GB budget, with a documented path to scale.
